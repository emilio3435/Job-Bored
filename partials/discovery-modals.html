You don’t need a webhook

The dashboard reads and writes your Pipeline sheet directly. A discovery webhook URL is only for the optional Run discovery button or tools that speak that contract.

  • Manual: paste or type jobs into the Sheet; they appear in Command Center after refresh.
  • Scheduled jobs: GitHub Actions, Apps Script triggers, n8n, or cron can add rows — no browser POST required.
  • Deep reading: docs/DISCOVERY-PATHS.md in the repo (diagrams and template links).

Set up a discovery webhook

Each person deploys their own discovery automation (this app does not host one). The dashboard-managed Google Apps Script path below is only a stub for webhook verification; it does not find jobs by itself.

Recommended real path: connect a user-owned agent, worker, or scheduled job that writes Pipeline rows. Start with integrations/openclaw-command-center/README.md or docs/DISCOVERY-PATHS.md. If that agent runs on your own machine, use first, then put the Worker URL in Settings.

Stub wiring path (first time ~15–30 min):
  1. 1. New script & paste code

    Open script.google.com → New project. Replace Code.gs with the file integrations/apps-script/Code.gs from your clone (or view raw on GitHub from the same path).

  2. 2. Script properties

    Gear icon → Script properties → add SHEET_ID (same ID as Spreadsheet URL or Sheet ID in Settings above). Optional: ENABLE_TEST_ROW = true for a test row on each POST.

  3. 3. Deploy as Web App

    Deploy → New deployment → type Web app → Execute as Me, Who has access Anyone (try first). Copy the URL ending in /exec.

  4. 4. Paste in Settings

    Put that URL in Discovery webhook URL here, save, then use Test webhook. This proves webhook wiring, but Run discovery stays disabled until a real discovery engine is connected.

CLI (clasp) — same project, local files

From the cloned repo, use the bundled npm scripts (no global clasp install): 

npm run apps-script:login
npm run apps-script:create
npm run apps-script:push

Then npm run apps-script:open and complete the first Web App deploy in the browser. Full walkthrough: integrations/apps-script/WALKTHROUGH.md.

CORS: the browser may show an error even when Apps Script ran. If Test webhook fails but npm run test:discovery-webhook succeeds from a terminal, schedule POSTs with GitHub Actions or use a Cloudflare Worker relay. That relay only fixes browser access; it does not turn the stub into a real discovery engine.

Local worker + ngrok

Use this when your real discovery engine runs on your laptop or homelab. The recommended local engine is the browser-use worker. The browser should not point straight at that local receiver. Instead use: dashboard → Cloudflare Worker → ngrok → local webhook.

Roles: the local webhook is the real discovery engine, the ngrok URL is its public HTTPS tunnel, and the Cloudflare Worker URL is the browser-facing URL you save in JobBored.
Fastest local path: from the repo root, run npm run discovery:bootstrap-local. It reuses the browser-use worker if it is already running, starts the browser-use worker by default, and writes a local JSON file so this modal can autofill on http://localhost. Advanced only: you can still point this path at Hermes/OpenClaw. 
npm run discovery:bootstrap-local

Start with your local discovery receiver.

Paste the exact local webhook URL first, then start ngrok on the same port.

  1. 1. Paste your local webhook URL

    Recommended example: http://127.0.0.1:8644/webhook.

    This is the real local discovery receiver. Recommended: browser-use worker. Advanced: Hermes / OpenClaw. Do not paste it into Discovery webhook URL on a hosted dashboard. 

    Paste your local webhook URL to generate /health.
  2. 2. Give it a public HTTPS tunnel with ngrok

    One-time auth, then start the tunnel on the same port as your local receiver:

    Need the token? Get it from dashboard.ngrok.com/get-started/your-authtoken. 

    ngrok config add-authtoken <YOUR_NGROK_TOKEN>
    ngrok http 8644

    Paste the https:// forwarding URL from ngrok, not the localhost line.

  3. 3. Use the generated public target in Cloudflare relay

    The Worker should forward to this ngrok URL plus your webhook path. The dashboard should still save the Worker URL, not this target directly. 

    Paste both the local webhook URL and the public ngrok URL to generate the target.

Cloudflare Worker relay

Use this when the browser should not POST directly to the real receiver, either because an Apps Script /exec URL lacks browser CORS headers or because your real discovery engine lives behind a public tunnel such as ngrok.

Fast path: let your coding agent run the repo helper script. The agent still needs your Cloudflare auth, but the helper now tries wrangler login automatically in an interactive terminal before stopping. If you use CLOUDFLARE_API_TOKEN plus CLOUDFLARE_ACCOUNT_ID, the helper can also reuse or create your account’s workers.dev subdomain automatically. With browser login only, the remaining one-time human step may be choosing that account subdomain in the terminal. Run the generated command from the repo root. Also use the open Worker URL for now: the dashboard does not send custom auth headers, so do not point it at /forward. Keep Cloudflare Access off for that open workers.dev URL.

Fastest path: let your coding agent deploy the relay.

The local helper script handles Wrangler deploy plus TARGET_URL secret upload, and the API-token path can also auto-create the one-time workers.dev account subdomain.

Advanced fallback: manual Wrangler steps

Use this only if you do not want to hand the repo to an agent.

These commands also assume you are starting from the repo root. 

cd templates/cloudflare-worker
wrangler login
wrangler secret put TARGET_URL
wrangler deploy

When Wrangler prompts for TARGET_URL, paste the target URL shown above.

Optional: restrict browser origin by adding the snippet below to templates/cloudflare-worker/wrangler.toml. 

[vars]
CORS_ORIGIN = "https://your-static-site.example"
https://your-static-site.example

If you do not have Wrangler installed yet, follow the Wrangler install guide.

Set up Run discovery

One-time setup: the dashboard POSTs to your HTTPS URL so your automation can run another discovery pass. There is no hosted discovery service from this app’s authors.

  1. Connect a real discovery engine that writes or updates Pipeline rows, such as the OpenClaw / agent guide or another worker you control.
  2. Copy the HTTPS webhook URL from that automation. If you only deploy the dashboard-managed Apps Script stub, treat it as wiring verification only.
  3. Paste it into Settings → Discovery webhook URL and save.

Each run sends a variation key and your discovery preferences so searches aren’t identical. Contract: AGENT_CONTRACT.md, SETUP.md, and docs/README.
