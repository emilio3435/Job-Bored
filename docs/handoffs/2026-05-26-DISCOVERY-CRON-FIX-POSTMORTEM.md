# Discovery Cron Fix — Postmortem (2026-05-26)

The daily discovery refresh cron (`com.jobbored.refresh` @ 08:15) had been silently failing for weeks with a layered cascade of HTTP errors in `state/launchd-refresh.log`: first `404 Not Found` (worker on the wrong port at install time), then `curl: Failed to connect to 127.0.0.1 port 8644` (the local discovery worker is not a persistent daemon — launchd does not keep it alive between cron fires, and any terminal session that started it would die at logout/sleep), then `"No stored candidateProfile found"` (someone tried `/discovery-profile?mode=refresh` instead of the actual `/webhook` route), and finally — after the worker was manually restarted — the steady-state failure: `HTTP 400 "Discovery intent cannot be blank when no target companies are configured."` because `worker-config.json` had `companies: []`, `targetRoles: []`, and `includeKeywords: []` (only `atsCompanies` was populated, which the webhook preflight does not consult). Fixed by (a) teaching `scripts/run-scheduled-discovery.mjs` to probe `/health` and auto-spawn the worker via `npm run discovery:worker:start-local` if it is down — detached, with output appended to `state/local-worker.log`, polling `/health` every 500 ms until ready or a 25 s deadline — so launchd no longer needs a separately-managed daemon, and (b) adding a cron-side `deriveIntentFromAtsCompanies()` fallback that flattens every `atsCompanies[*].roleTags` entry into the payload's `discoveryProfile.targetRoles` whenever the user has not supplied explicit `targetRoles`/`includeKeywords`, so the preflight's blank-intent guard passes without us having to touch the security/order invariant in `handle-discovery-webhook.ts`. Verified: `node scripts/run-scheduled-discovery.mjs --trigger scheduled-local --sheet-id … --port 8644` returns `HTTP 202 accepted_async runId=run_bed654b0eb0f4fd89e55a5aabc9f3700` from a cold worker (auto-start kicks in, becomes healthy in ~1 s, webhook accepts), and `launchctl start com.jobbored.refresh` produces the same `scheduled-discovery: accepted HTTP 202` line at the end of the launchd log — replacing the previous failure pattern.
